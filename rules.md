# BettaFish 项目排错与最佳实践规则

本文件记录了在开发与部署 BettaFish 舆情分析系统时，遇到的各类常见问题及其避免方式。请在后续开发中严格遵守。

## 1. 数据库与容器网络连接规范
**问题现象**：Insight Engine 报错 `[Errno 111] Connect call failed ('127.0.0.1', 5444)`，并且疯狂打印堆栈。
**问题原因**：
当代码运行在 Docker 容器内部时，如果 `.env` 中的 `DB_HOST` 配置为 `localhost` 或 `127.0.0.1`，代码会去尝试连接**当前容器本身**的 5444 端口，而不是去连接名为 `db` 的 PostgreSQL 容器，从而导致连接被拒。
**最佳实践**：
- **动态探测机制**：在数据库连接初始化的工具类中（如 `utils/db.py`），应当自动检测是否运行在 Docker 中（例如检查是否存在 `/.dockerenv` 文件）。如果是，则需要将指向 `localhost` 的连接地址自动重写为 Docker Compose 网络中的服务名（如 `db`）及内部端口（如 `5432`）。
- **精简日志输出**：在事件循环或高频触发的组件（如 `run_until_complete(fetch_all)`）中，捕获异常时切忌使用 `logger.exception()`，而应当使用 `logger.error()` 打印精简的错误信息。否则一旦数据库掉线，海量的错误堆栈会迅速掩盖掉所有有用的日志。

## 2. 大模型“幻觉（Hallucination）”与数据编造的防范
**问题现象**：当大模型在搜索全网或数据库时未能找到与用户查询强相关的真实数据时，为了强行满足 Prompt 中“提供代表性评论”、“生成字数”等要求，它会“一本正经地胡说八道”，凭空捏造虚假的新闻事件、媒体报道、具体的网民 ID（如“@游戏达人阿哲 转发4219次”），甚至是科幻/伪科学名词（如将未发售的游戏描述为“基于ΔW≥0.372神经公理的认知基础设施”）。
另一种隐蔽的幻觉是“**张冠李戴（URL Mismatch）**”：大模型会凭空捏造一个极具权威感的学术标题（如《苯教水祭载体白皮书》），然后为了通过系统对“必须包含真实 URL”的硬性校验，强行把这个假标题绑定到一个真实存在但内容毫不相干的网页链接上（如某个游戏网站的盘点文章）。
**问题原因**：
LLM 本质上是基于概率的文字接龙引擎，在缺乏事实锚点且被硬性任务指令逼迫时，其生成虚假信息的概率会呈指数级上升。当系统加入了“必须包含真实链接”的强制校验后，大模型会为了“交差”而强行进行拼凑。
**最佳实践（防幻觉护栏）**：
- **项目底线原则（闭嘴原则）**：在所有直接处理搜索结果的 Agent Prompt 中，必须在最高权重层级（System Prompt 或 User Prompt 开头）加入极其严格的防幻觉护栏：
  > “如果搜索结果中没有相关数据，你必须明确回复‘未检索到相关数据’或‘数据不足，无法分析’，**绝对禁止**自行编造数字、用户ID、评论、新闻事件或学术名词。你所有的引用必须 100% 来源于提供的搜索结果。如果没有搜索到任何有效信息，请直接按照实际情况反馈，告知报告无法生成，而不是胡编滥造。”
- **严禁科幻/学术化臆想与张冠李戴**：对于游戏、娱乐或常规商业查询，绝不可凭空捏造科幻设定、伪科学术语。必须明确告知大模型：**提取的 URL 必须与其真实的上下文和真实的网站标题完全一致，绝对禁止将伪造的高级标题绑定到无关的真实 URL 上。**
- **杜绝字数强制指标**：绝对不要在 Prompt 中设定硬性的字数下限（如“不少于1000字”）。大模型为了凑字数，在无话可说时必定会开始编造废话或虚假事实。应改用“内容详实”、“结构深入”等质量层面的引导词。

## 3. 大模型 JSON 输出截断的防御机制
**问题现象**：`JSON解析失败: Unterminated string starting at...`，随后导致状态无法更新。
**问题原因**：
部分大模型（尤其是依赖第三方聚合 API 时）具有严格的单次输出 `max_tokens` 限制。当遇到如“生成上千字的长篇分析报告”这类任务时，模型经常输出到一半就被强行掐断，导致生成了一个尾部不闭合的残缺 JSON。
**最佳实践**：
- **放弃手写正则修复**：永远不要尝试用手动写正则、计算花括号/方括号数量的方式去修复 JSON（因为极难处理值被截断的情况）。
- **引入专业库**：在所有涉及解析 LLM 输出 JSON 的地方，必须使用 `json_repair` 库的 `repair_json(text)` 进行兜底。该库能够智能推断结构并自动补全残缺的引号和括号，极大提升解析的健壮性。
- **配置最大 token 数**：在向模型发请求的底层驱动（如 `llms/base.py`）中，必须显式传递并拉高 `max_tokens`（如 4096 或 8192），尽量避免模型因字数限制被截断。

## 4. 大模型调用超时与反思死循环的防御
**问题现象**：模型报错 `Error code: 403` 或 `429 Rate Limit`。
**问题原因**：国内网络直接调用海外免费聚合 API（如 OpenRouter）易受限，或因为高频次的反思（Reflection）循环触发限流。
**最佳实践**：
- **降级与切换机制**：在配置文件中始终准备国内可用的稳定模型（如阿里云 Qwen）。
- **提示词防发散**：在编写 prompt 时，必须明确搜索词（Query）的提炼规则（如：2-4个核心关键词），避免模型把一长串句子当做搜索词丢给 Tavily API 从而引发超时。
- **避免硬性字数要求**：在 prompt 中禁止出现“不少于1000字”、“新增内容100%”等字数下限要求。这不仅容易导致 JSON 截断，还会迫使模型在无有效信息时产生“幻觉（Hallucination）”。应改用“详实”、“多维度”等质量向的引导词。

## 5. 新旧功能兼容与平滑演进原则
**问题现象**：在引入新功能（如增加新的搜索 API、修改任务 ID 生成规则等）时，采取“一刀切”的破坏性更新，导致旧环境报错、历史数据无法读取或用户原有的配置失效。
**问题原因**：
在重构或增加新功能时，缺乏向下兼容的意识，直接移除了旧的配置字段或改变了核心数据结构的解析逻辑。
**最佳实践**：
- **配置兼容（Fallback）**：如果在配置文件（如 `.env`、`config.py`）中新增或修改了配置项，代码层面必须允许旧配置项继续生效（例如通过 `or` 逻辑或备用判断，回退到旧字段）。
- **逻辑与数据兼容**：对于任务 ID、报告路径等核心机制的修改，必须考虑到存量历史数据的读取。需在解析中加入兼容分支（例如尝试按新格式解析，失败则按旧格式读取），确保历史记录随时可用。
- **保留用户选择权**：在涉及多种模式（如搜索引擎切换）的重构中，应通过代码底层的智能路由去适配新增的并行逻辑，保留并尊重用户原有的配置习惯和选择控制权。

## 6. 大模型上下文超限与死循环重试的防御
**问题现象**：报错 `processing failed: request was too large` 或 `context length exceeded`。随后 `retry_helper` 触发，在 60 秒后再次尝试，由于 Prompt 长度并未改变，导致陷入无限失败的死循环。
**问题原因**：
当聚合了三个子引擎的报告、长篇网页正文以及大量评论数据后，组合出的 `user_prompt` 长度突破了 LLM（如 Qwen/Claude 等）允许的最大上下文窗口（Context Window）。
**最佳实践**：
- **动态递减截断重试**：在底层封装调用（如 `ReportEngine/llms/base.py` 的 `stream_invoke`）中，不仅需要设置一个绝对安全的初始上限（如 100,000 字符），更要在捕获到 `request was too large` 异常时，在方法**内部**利用 `while` 循环直接进行降级重试。
- **保护核心信息**：截断策略应采用“保留头尾，砍掉中间”（`user_prompt[:half] + "...[截断]..." + user_prompt[-half:]`）的方式。因为 Prompt 的开头通常是系统指令或核心任务，结尾通常是最新补充的上下文或输出格式要求，中间部分往往是可舍弃的冗长数据。
- **阻断外部死循环**：通过在异常捕获内部将 `user_prompt` 缩减 30% 并 `continue` 重新发起请求，彻底避免将超长异常抛给外层的 `@with_retry` 装饰器，从而打断毫无意义的等候和死循环。

## 7. 前后端状态同步与激进的自动触发机制防范
**问题现象**：用户在前端点击了“取消生成报告”，任务成功停止。但当用户“刷新页面”或“切换Tab”后，系统又像幽灵一样自动重新开始生成报告。
**问题原因**：
前端在初始化或定时轮询时，如果检测到后端返回了 `engines_ready: true`（文件已就绪），就包含了一段**过于激进的自动生成逻辑（Auto-Generate）**。由于页面刷新会清空前端内存（如 `autoGenerateTriggered` 标志位），导致前端“忘记”了用户刚刚执行的取消操作，再次看到文件就绪就自作主张地触发了生成。
**最佳实践**：
- **摒弃激进的自动执行逻辑**：对于耗时极长、消耗极大资源（如大模型调用、大规模并发检索）的任务，**绝对不要使用自动触发机制（Auto-Start/Auto-Generate）**。必须将控制权完全交还给用户，要求用户必须通过明确的物理点击（Click）来触发。
- **状态机的边界测试（Edge Case Analysis）**：在开发任何状态控制（如取消、暂停）功能时，必须将“页面刷新（F5）”、“网络断开重连”、“Tab页来回切换”作为必测的边界条件，确保后端的持久化状态与前端的无状态重载之间不会发生逻辑冲突。

## 8. 复杂重构的执行闭环与“言行一致”审查
**问题现象**：在进行多模块的复杂系统设计或重构时（如引入“AI记忆与切片机制”），开发助手或工程师在语言方案上给出了完美的设计，但在实际修改代码时却“言行不一”——只实现了部分工具函数，却遗漏了最关键的数据注入或上层调用，导致机制未真正生效（半成品）。
**问题原因**：
大语言模型（LLM）或开发者在处理超长上下文和跨文件修改时，容易产生“注意力漂移（Attention Drift）”和“伪完成幻觉”。在修改完第一部分代码后就产生满足感，误以为整个大任务已经闭环，缺乏回溯审查整个承诺清单的机制。
**最佳实践**：
- **强制 TodoList 拆解与追踪**：在动手修改前，必须将高维度的架构设计拆解为具体到文件的修改清单（如：1.工具函数实现 2.状态结构修改 3.调用点参数注入）。
- **交付前的 Self-Check（自我审查）**：在提交代码（交付任务）前，必须强制进行一次内部复盘代码审查：“我刚刚在方案中承诺的所有机制，是否都已经 100% 转化为了真实的、被正确调用的代码修改？”
- **端到端（End-to-End）闭环要求**：绝不交付“半成品”。不仅要写好底层逻辑，还要确保数据流从输入到输出的完整闭环（例如：写了数据过滤算法，就必须确保在 Payload 组装时真正替换了旧变量并调用了该算法）。

## 9. 避免 Flask 路由重名覆盖与后台线程的真正中断
**问题现象**：用户点击“取消生成”，前端显示任务已取消，但后台大模型仍在继续生成。当用户刷新页面并点击历史记录时，进度条“死灰复燃”继续走动（幽灵重启）。
**问题原因**：
1. **路由函数同名覆盖**：Flask 中定义了两个 `@report_bp.route('/cancel/<task_id>', methods=['POST'])` 路由，分别绑定在 `cancel_report` 和 `cancel_task` 函数上。后定义的函数覆盖了前者的逻辑，导致真正能向后台线程注入中断异常（`CancelGenerationException`）的逻辑未被执行。
2. **状态更新与线程解绑的竞态**：取消接口中虽将状态标为 `cancelled`，但由于后台流式事件（SSE）仍在运行，每一次 `yield` 都将状态强制刷回 `running`，导致前端轮询接口时看到任务还在运行。
**最佳实践**：
- **清理冗余路由**：在添加新的 REST 接口时，必须先全局搜索是否有相同的 `route` 路径。严禁在一个 Blueprint 中对同一路径绑定多个处理函数。
- **确保异常透传与线程安全**：在中止长耗时后台线程（如大模型生成）时，必须通过标志位（如 `task.is_cancelled = True`）配合抛出专属的控制流异常（如 `CancelGenerationException`）来安全退出，并且在捕获该异常时，避免进行退避重试（Retry），直接释放全局锁并清空当前任务状态（`current_task = None`）。

## 10. 爬虫数据源与分析引擎的严格边界隔离
**问题现象**：在引入新的爬虫数据源（如基于 Playwright 的 MediaCrawler）时，为了让新数据被大模型识别，开发者直接去修改 `InsightEngine` 或 `MediaEngine` 的核心检索逻辑或大模型 Prompt，导致系统架构变得极度耦合。
**问题原因**：
混淆了“获取数据”与“消费数据”的边界。BettaFish 的架构设计原则是**读写分离**。
**最佳实践**：
- **爬虫引擎（Crawler Engine / Data Ingestion）**：只能负责对外的“脏活累活”（反爬、代理、浏览器模拟）。新爬虫抓取到的数据必须统一汇入本地数据库（如 PostgreSQL），并**在入库的瞬间或之后立即由数据接入层计算并填补 `embedding` 向量**。
- **本地数据库（护城河）**：它是所有数据的最终着陆点，三大引擎只认本地数据库中的数据。
- **三大分析引擎（Insight / Media / Query Engine）**：严禁直接触碰互联网。它们的职责仅仅是通过向量搜索（Cosine Similarity）和 SQL 查询，从本地数据库中提取知识并进行深度分析。引入新爬虫数据源时，**绝对不需要且不允许修改三大引擎的核心工作模型**。

## 11. Playwright/MediaCrawler 的容器化与登录态持久化
**问题现象**：使用基于无头浏览器（如 Playwright）的爬虫时，每次抓取都提示需要扫码登录，导致无人值守的自动化流水线中断。
**问题原因**：
每次启动 Docker 容器或重新执行 Python 脚本时，如果没有指定持久化的 User Data 目录，Playwright 会启动一个全新的、纯净的浏览器上下文，所有 Cookie 和登录态（LocalStorage）均会丢失。
**最佳实践**：
- **挂载持久化目录**：必须在 `MediaCrawler` 或类似爬虫工具的配置中，开启 `user_data_dir` 选项，并将该目录通过 `docker-compose` `volumes` 挂载到宿主机（如 `./data/browser_data:/app/browser_data`）。
- **人工扫码初始化**：在新部署系统时，提供一个专门的初始化脚本，以有头模式（`headless=False`）运行一次，让人工扫码登录各平台（小红书、抖音等），登录态被写入持久化目录后，后续的 Docker 自动化任务即可永久复用这些 Cookie，实现无感爬取。
